# 客观题批改软件产品化改造计划

## 1. 当前阶段重新定位

当前阶段先不做题库联动，先把系统做成一个真正合格、稳定、易用、老师能独立完成操作的“客观题批改软件”。

近期主链目标是：

选择班级 -> 新建考试 -> 录入标准答案 -> 采集学生作答 -> 确认作答 -> 批改前检查 -> 规则判分 -> 班级简报 -> 详细分析 -> Excel 导出。

优先级需要准确区分：

- 题库接入后置。
- AI 讲评后置。
- 自动组卷后置。
- 复杂图表后置。
- 真实 OCR 模型接入可以后置。
- OCR 数据链路、作答采集、识别草稿、人工确认、判分算法、填空题复核机制必须前置设计。

原因很简单：客观题批改软件的地基不是“有没有 AI”，而是“答案怎么进入系统、作答如何确认、分数怎么算、异常怎么处理、老师能否复核”。如果没有这些底座，后面接任何 OCR 或题库都会把不稳定数据推入成绩。

当前阶段的判断：

- 先做独立批改软件合格。
- 先确保老师能自己完成一次批改。
- 先确保单选、多选、判断题、填空题的判分规则和测试稳定。
- 先设计作答采集和 OCR 草稿确认链路。
- 先确保判分结果包含状态、原因、复核标记，而不只是分数。
- 暂缓题库、AI、自动组卷、复杂图表和跨系统耦合。

## 2. 目标用户与真实使用场景

### 不懂电脑的老师

这类老师最怕的是不知道从哪里开始、看不懂按钮、遇到错误不知道怎么办。他们希望系统像向导一样告诉自己：

1. 先选哪个班；
2. 再填考试名称；
3. 然后把正确答案放进去；
4. 再把学生答案放进去；
5. 检查系统提示的问题；
6. 点“开始批改”；
7. 看班级情况；
8. 导出成绩表。

对他们来说，系统最大的价值不是“分析多高级”，而是“不迷路、不出错、能看懂、能导出”。

### 会电脑的老师

这类老师愿意看更细的数据，关注效率和可追踪性。他们希望：

1. 快速导入学生名单；
2. 快速粘贴或导入答案；
3. 快速粘贴、导入或采集学生答案；
4. 快速发现异常答案、空答、低置信度识别结果；
5. 快速批改；
6. 快速导出 Excel；
7. 快速查看错题分布、知识点薄弱和学生个人问题。

### 批改后准备讲评课的老师

老师批改完后最关心：

1. 哪几道题错得最多；
2. 哪些题不需要讲；
3. 哪些知识点普遍薄弱；
4. 哪些学生需要单独关注；
5. 讲评课应该先讲什么。

当前报告要先服务这个场景：让老师能直接拿“班级简报”和“讲评建议”备课。

## 3. 合格批改软件的最低标准

### 输入合格

必须支持：

1. 学生名单导入；
2. 学生名单手动编辑；
3. 标准答案手动录入；
4. 标准答案批量粘贴；
5. 标准答案表格导入；
6. 学生答案手动录入；
7. 学生答案 Excel / CSV 导入；
8. 学生答案批量粘贴；
9. 图片上传、手机拍照、摄像头拍照的作答采集链路设计；
10. OCR 草稿人工确认机制；
11. 空答处理；
12. 异常选项提示；
13. 填空题待复核处理。

当前完成度判断：

- 学生名单导入已具备，`roster_manager.py` 支持 CSV/XLSX。
- 标准答案导入已有基础，`app/data_io.py` 支持 CSV/XLSX/DOCX/PDF/TXT/图片入口和确认草稿。
- 学生答案批量导入已有基础，`web_app.py` 支持上传并转换表格。
- 拍照保存入口已有雏形，但真实 OCR 和确认链路尚未产品化。
- 手动编辑学生名单、手动录入学生答案、批量粘贴作答还需要补齐。

### 判分合格

必须支持：

1. 单选题；
2. 多选题；
3. 判断题；
4. 填空题；
5. 空答；
6. 异常答案；
7. 多选部分得分；
8. 填空题等价判断；
9. 填空题人工复核；
10. 修改答案后重新计算；
11. 修改学生作答后重新判分。

当前完成度判断：

- 单选、多选、空答、异常答案、多选部分得分已有核心规则。
- 判断题没有独立产品化题型，需要明确 T/F、√/×、对/错 等价输入。
- 填空题已有别名和容差的局部能力，但还不是完整数学填空判分算法。
- 判分结果需要从“分数 + 状态”升级为“分数 + 状态 + 原因 + 是否需复核 + 来源 + 置信度”。

### 检查合格

批改前必须检查：

1. 是否有班级；
2. 是否有学生名单；
3. 是否有考试；
4. 是否有标准答案；
5. 是否有学生作答；
6. 题号是否连续；
7. 题数是否一致；
8. 是否有空答；
9. 是否有异常选项；
10. 是否有 OCR 低置信度结果；
11. 是否有未确认 OCR 草稿；
12. 是否有填空题待复核；
13. 是否有重复学生；
14. 是否有缺失姓名；
15. 是否有分值异常；
16. 是否有无法判分的答案。

检查结果要区分：

- 阻止批改的问题；
- 允许继续但需要提醒的问题；
- 必须人工确认的问题。

### 输出合格

必须输出：

1. 学生总分；
2. 每题得分；
3. 每题作答；
4. 判分状态；
5. 判分原因；
6. 是否需要复核；
7. 错题号；
8. 每题正确率；
9. 班级平均分；
10. 最高分；
11. 最低分；
12. 错得最多的题；
13. 薄弱知识点；
14. Excel 成绩表；
15. 简洁班级报告。

### 使用体验合格

必须做到：

1. 按钮名字简单；
2. 步骤清楚；
3. 报错能看懂；
4. 不懂电脑的人也能照着做；
5. 会电脑的人能看到详细数据；
6. 不强迫老师理解技术术语；
7. 不让老师重复输入；
8. 不轻易丢失数据；
9. OCR 草稿不自动进入成绩；
10. 填空题不确定时让老师复核。

### 已经够用或接近够用的功能

- 本地网页端已经存在，老师不必只用命令行。
- 班级导入和班级索引已经存在。
- 考试创建流程已有网页向导。
- 标准答案导入来源较多，包括表格、文本、Word/PDF、图片占位入口。
- 学生作答表格导入已经存在。
- 批改前预览和数据检查已经存在。
- 自动判分核心已经存在。
- 多选部分得分规则已经实现。
- 班级统计、题目分析、知识点分析、讲评计划、Excel 报告已经有雏形。
- 测试入口 `python run_tests.py` 已覆盖一批核心规则和文件流程。

### 还不够老师真实使用的功能

- 学生名单手动新增、修改、删除不够产品化。
- 标准答案手动录入虽然有草稿表格，但对不懂电脑老师仍可能偏复杂。
- 学生答案手动录入或粘贴录入能力不够清晰。
- OCR / 图片 / 拍照作答链路还停留在入口层，需要补“草稿 -> 确认 -> 正式作答”的模型。
- 判断题作为独立题型的展示、录入、判分和测试不够明确。
- 填空题判分算法不够完整，不能只靠字符串匹配。
- 批改前异常检查需要覆盖 OCR 低置信度、未确认草稿、填空题待复核。
- 班级简报还需要更聚焦，先给结论。
- 学生个人情况需要更像老师能直接看懂的个人报告。
- Excel 导出需要增加复核表和原始作答追溯。

## 5. 批改主流程设计

推荐最简单主流程：

选择班级 -> 新建考试 -> 录入答案 -> 采集学生作答 -> 确认学生作答 -> 检查数据 -> 开始批改 -> 查看报告 -> 导出成绩。

### 选择班级

页面文案：“这次批改哪个班？”

动作：

- 选择已有班级；
- 新建班级；
- 导入学生名单；
- 查看学生名单是否正确。

### 新建考试

页面文案：“这次考试叫什么？”

字段：

- 考试名称；
- 考试日期；
- 科目；
- 总分，可选；
- 备注，可选。

### 录入答案

页面文案：“把正确答案放进来。”

默认方式：

- 手动输入答案；
- 批量粘贴答案；
- 从 Excel 导入。

题库导入暂缓，不作为当前主线。

### 采集学生作答

页面文案：“把学生作答放进来。”

来源包括：

1. 手动录入；
2. Excel 导入；
3. 批量粘贴；
4. 图片上传；
5. 手机拍照；
6. 摄像头拍照；
7. 答题卡识别；
8. 普通试卷作答痕迹识别，后期再考虑。

当前可以先实现手动、Excel、粘贴的确认模型；真实 OCR 模型接入可以后置，但数据结构和确认流程必须前置。

### 确认学生作答

所有非人工直接录入的数据都应先进入“作答草稿”。老师确认后，才转为正式学生作答。

页面应显示：

- 学生；
- 题号；
- 识别答案；
- 置信度；
- 是否低置信度；
- 是否空答；
- 是否异常；
- 是否需要复核；
- 原图引用，可选。

### 检查数据

页面文案：“先检查有没有问题。”

显示：

- 学生人数；
- 题目数量；
- 标准答案是否完整；
- 学生答案是否完整；
- 异常选项；
- 空答；
- OCR 低置信度；
- 未确认草稿；
- 填空题待复核；
- 未匹配学生；
- 是否可以开始批改。

### 开始批改

主按钮：“开始批改”。

批改过程只能调用规则判分。OCR 不能直接生成最终分数，AI 不能参与判分。

### 查看报告

默认打开“班级简报”。

二级入口：

- 查看详细分析；
- 查看学生情况；
- 查看复核项；
- 导出成绩表。

## 8. OCR 与作答采集设计

### 作答来源

学生作答来源应统一进入“作答采集层”：

1. 手动录入；
2. Excel 导入；
3. 批量粘贴；
4. 图片上传；
5. 手机拍照；
6. 摄像头拍照；
7. 答题卡识别；
8. 普通试卷作答痕迹识别，后期再考虑。

不同来源只影响 `source` 和 `confidence`，不应改变最终判分规则。

### OCR 流程

图片 / 拍照 -> OCR 识别 -> 生成作答草稿 -> 标记低置信度项目 -> 老师确认或修改 -> 转为正式学生作答 -> 规则判分。

必须强调：

- OCR 不能直接进入最终成绩；
- OCR 不能覆盖老师已确认的数据；
- OCR 不能修改标准答案；
- OCR 不能直接生成最终分数；
- OCR 结果必须保留原图追溯；
- 低置信度结果必须高亮；
- 老师确认后的数据才是正式作答。

### OCR 识别状态

建议状态：

- `draft`：识别草稿；
- `confirmed`：已确认；
- `low_confidence`：低置信度；
- `conflict`：冲突；
- `blank`：空答；
- `invalid`：异常答案；
- `needs_review`：需要老师确认。

给老师看的提示必须是自然语言：

- “张三第 8 题识别不清，请确认。”
- “李四第 12 题可能为空答。”
- “王五第 3 题出现异常选项 E。”
- “第 15 题是填空题，有 6 个答案需要人工复核。”

不要把技术状态码直接展示给老师。

### 作答草稿数据建议

草稿至少应包含：

- studentId；
- studentName；
- questionNumber；
- rawAnswer；
- normalizedAnswer；
- source；
- confidence；
- status；
- originalImageRef；
- confirmedBy；
- confirmedAt。

## 9. 选择题 / 多选题 / 判断题判分算法

判分规则不能写在 UI，必须在独立判分模块中实现，并且必须有测试。

### 单选题

规则：

- 标准答案只有一个；
- 学生答案完全一致得满分；
- 空答得 0；
- 多选或非法选项提示异常并得 0；
- 修改答案后可重新判分。

判分状态：

- correct；
- wrong；
- blank；
- invalid。

### 多选题

用户确认规则：

6 分题，正确答案 3 个：

- 选对 1 个得 2 分；
- 选对 2 个得 4 分；
- 全对得 6 分；
- 只要含错误选项得 0；
- 空答得 0。

6 分题，正确答案 2 个：

- 选对 1 个得 3 分；
- 全对得 6 分；
- 只要含错误选项得 0；
- 空答得 0。

一般化规则：

如果满分 `fullScore`，正确答案数量 `correctCount`，学生选对数量 `selectedCorrectCount`：

- 空答：0；
- 包含错误选项：0；
- 学生选项全部属于正确答案集合：`score = fullScore * selectedCorrectCount / correctCount`；
- 全对：满分。

分数取整策略必须明确。当前高中数学客观题一般按整数处理；如果出现不能整除，应在试卷配置中明确保留小数或四舍五入，默认不要静默取整。

### 判断题

规则：

- 支持 T/F、√/×、对/错、正确/错误 等价输入；
- 匹配得满分；
- 空答得 0；
- 非法答案提示异常并得 0；
- 等价映射应在判分模块统一处理。

## 10. 填空题判分算法设计

填空题是数学批改系统的关键难点，不能简单写成“字符串相同即可”。

填空题可能包括：

1. 整数；
2. 小数；
3. 分数；
4. 根式；
5. 区间；
6. 集合；
7. 坐标；
8. 向量；
9. 表达式；
10. 多空答案；
11. 多个等价答案；
12. 单位或文字说明。

填空题判分应优先使用规则算法：

- 去空格；
- 中英文符号统一；
- 分数标准化；
- 小数容差；
- 根式标准化；
- 集合顺序无关；
- 区间格式统一；
- 多答案集合匹配；
- 多空逐空判定；
- 必要时人工复核。

AI 不能作为填空题判分主算法。AI 最多只能作为讲评或辅助解释，不能直接决定得分。

填空题判分状态必须包括：

- `correct`：确定正确；
- `wrong`：确定错误；
- `partial`：部分正确；
- `needs_review`：需要人工复核；
- `invalid`：格式异常；
- `blank`：空答。

示例：

- 标准答案：`1/2`；学生答案：`0.5`；应判正确。
- 标准答案：`√2`；学生答案：`根号2`；应考虑标准化后判正确。
- 标准答案：`{1,2}`；学生答案：`{2,1}`；应判正确。
- 标准答案：`(1,2)`；学生答案：`1,2`；是否正确取决于题目要求，可能需要人工复核。
- 标准答案：`x>1`；学生答案：`(1,+∞)`；可能等价，但需要表达式或区间等价判断；算法不能确定时应进入人工复核。

填空题优先策略：

1. 明确可判等价的，自动判分；
2. 明确错误的，自动判错；
3. 可能等价但算法不确定的，进入 `needs_review`；
4. 格式异常的，进入 `invalid` 或 `needs_review`；
5. 老师复核后的结果可重新计算总分。

## 11. 判分结果数据结构

判分结果不能只是数字。建议结构包括：

- questionNumber；
- questionType；
- studentAnswer；
- normalizedAnswer；
- correctAnswer；
- score；
- maxScore；
- status；
- reason；
- needsReview；
- confidence；
- source；
- originalImageRef，可选。

确定正确示例：

```json
{
  "questionNumber": 12,
  "questionType": "blank",
  "studentAnswer": "0.5",
  "normalizedAnswer": "1/2",
  "correctAnswer": "1/2",
  "score": 5,
  "maxScore": 5,
  "status": "correct",
  "reason": "分数与小数等价",
  "needsReview": false
}
```

不确定示例：

```json
{
  "questionNumber": 13,
  "questionType": "blank",
  "studentAnswer": "x>1",
  "correctAnswer": "(1,+∞)",
  "score": null,
  "maxScore": 5,
  "status": "needs_review",
  "reason": "可能等价，但需要人工确认",
  "needsReview": true
}
```

产品规则：

- `needsReview=true` 的题目不能被当作最终成绩静默导出。
- 老师复核后应记录复核结果和复核时间。
- 总分计算必须明确是否包含待复核题。建议默认阻止最终批改，允许“试算报告”但必须标记。

## 12. 批改前异常检查设计

批改前检查应分为“阻止批改”“允许继续但提醒”“必须人工确认”。

### 阻止批改

- 没有班级；
- 没有学生名单；
- 没有考试；
- 没有标准答案；
- 没有学生作答；
- 标准答案题号重复；
- 题数严重不一致；
- 分值为空或非法；
- 存在无法判分且未复核的答案。

### 允许继续但提醒

- 某学生有空答；
- 某题空答人数较多；
- 作答文件中出现标准答案没有的题号；
- 学生姓名缺失但学号存在；
- 班级名单中有学生没有作答。

### 必须人工确认

- OCR 低置信度结果；
- 未确认 OCR 草稿；
- 填空题待复核；
- 作答与老师已确认数据冲突；
- 填空题可能等价但算法不确定。

老师看到的提示示例：

- “第 3 题没有标准答案，请补充后再批改。”
- “张三第 8 题识别不清，请确认。”
- “李四第 12 题可能为空答。”
- “王五第 3 题出现异常选项 E。”
- “第 15 题是填空题，有 6 个答案需要人工复核。”

## 15. 工程结构风险

### legacy

`legacy/objective_grader_legacy.py` 仍承担模型、判分、统计、报告、HTML、Excel、校验等大量职责。继续把核心功能加进去会让系统越来越难维护。近期应避免新增 legacy 核心逻辑。

### app/core.py

`app/core.py` 当前主要 re-export legacy 中的 `AnswerKey`、`QuestionSpec`、`StudentResult`、`score_answer`、`grade_all` 等。它还不是真正独立的判分核心。

### app/workflow.py

`app/workflow.py` 是当前批改流程编排中心，负责读取、判分、校验、生成报告、归档等。它对产品闭环很关键，但职责偏重。后续应逐步拆出 use case、report、exporter。

### web/static/app.js

`web/static/app.js` 同时承担全局状态、API 调用、DOM 渲染、页面导航、摄像头、文件上传、表格编辑、批改流程控制、报告展示。继续扩展 UI 会让风险增加。后续应拆成 state、api、wizard、tables、reports、camera 等模块。

### 报告生成

报告生成散布在 legacy 和 `app/workflow.py` 中，HTML、CSV、Excel 混在流程里。后续产品化报告时，应先定义报告数据模型，再由不同 exporter 输出 HTML/Excel/CSV。

### Excel 导出

当前已有 Excel 输出能力，但需要稳定面向老师的 Sheet 结构。Excel 导出不应直接依赖零散中间 CSV 文件名，而应依赖稳定的报告数据对象。

### 测试覆盖

当前 `python run_tests.py` 覆盖 13 个测试，已有基础。但产品化前还需要补：

- 判断题；
- 多选 3 个正确答案的部分得分；
- 填空题标准化；
- 填空题等价判断；
- 填空题 `needs_review`；
- OCR 草稿状态；
- 低置信度检查；
- 修改答案后重新判分；
- 异常答案提示；
- Excel Sheet 字段；
- 班级简报数据；
- 学生个人情况数据。
