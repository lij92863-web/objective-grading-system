# 客观题批改软件产品化改造计划

## 1. 当前阶段重新定位

当前阶段的第一目标不是题库联动、OCR、AI 或复杂分析大屏，而是先把系统做成一个真正合格、稳定、易用、老师能独立完成操作的“客观题批改软件”。

近期产品定位应收窄为：

学生名单 -> 新建考试 -> 录入标准答案 -> 导入学生作答 -> 批改前检查 -> 自动判分 -> 班级简报 -> 详细分析 -> Excel 导出。

题库系统连接仍是长期方向，但当前只需要在架构上不堵死，不要现在实现 `paper-package.json`、`assessment-result.json`、题库回流、自动组卷、OCR、AI。否则主链还没有足够稳，就会把不确定功能接进核心流程，增加老师使用失败率和代码维护风险。

当前阶段的判断：

- 先做“独立批改软件合格”。
- 先确保老师能自己完成一次批改。
- 先确保判分、检查、报告、导出稳定。
- 暂缓所有跨系统和智能化增强。

## 2. 目标用户与真实使用场景

### 不懂电脑的老师

这类老师最怕的是不知道从哪里开始、看不懂按钮、遇到错误不知道怎么办。他们希望系统像一个向导一样告诉自己：

1. 先选哪个班；
2. 再填考试名称；
3. 然后把正确答案放进去；
4. 再把学生答案放进去；
5. 点“开始批改”；
6. 看班级情况；
7. 导出成绩表。

对他们来说，系统最大的价值不是“分析多高级”，而是“不出错、不迷路、能导出、能看懂”。

当前产品应重点满足：

- 首页有明确的“新建一次批改”入口；
- 每一步只出现当前需要做的事；
- 不出现 `manifest`、`submission`、`analytics`、`schema` 等术语；
- 错误提示必须变成老师能理解的话；
- 批改完成后先显示班级简报，而不是复杂表格。

### 会电脑的老师

这类老师愿意看更细的数据，关注效率和可追踪性。他们希望：

1. 快速导入学生名单；
2. 快速粘贴或导入答案；
3. 快速导入学生作答；
4. 批改前自动发现异常；
5. 批改后马上看每题正确率、错题分布、知识点薄弱；
6. 导出详细 Excel；
7. 根据数据安排讲评和补救。

对他们来说，系统不能只给一个总分表，还要给“每题、每人、每个知识点”的明细。

### 批改后准备讲评课的老师

这类场景发生在考试结束后，老师最关心：

1. 哪几道题错得最多；
2. 哪些题大部分学生都会，不必细讲；
3. 哪些知识点普遍薄弱；
4. 哪些学生需要单独关注；
5. 讲评课应该先讲什么。

当前系统已经有讲评计划、错题列表、知识点画像等雏形，但还需要把它们组织成老师一眼能用的“班级简报”和“讲评建议”。

## 3. 合格批改软件的最低标准

### 输入合格

必须支持：

1. 学生名单导入；
2. 学生名单手动编辑；
3. 标准答案手动录入；
4. 标准答案批量粘贴；
5. 标准答案表格导入；
6. 学生答案手动录入；
7. 学生答案批量导入；
8. 空答处理；
9. 异常选项提示。

当前完成度判断：

- 学生名单导入已具备，`roster_manager.py` 支持 CSV/XLSX。
- 标准答案导入已具备，`app/data_io.py` 支持 CSV/XLSX/DOCX/PDF/TXT/图片入口和确认草稿。
- 学生答案批量导入已具备，`web_app.py` 支持上传并转换表格。
- 手动编辑学生名单、手动录入学生答案还不够产品化，需要后续补齐。

### 判分合格

必须支持：

1. 单选题；
2. 多选题；
3. 判断题；
4. 空答；
5. 异常答案；
6. 多选部分得分；
7. 修改答案后重新计算；
8. 修改学生作答后重新判分。

当前完成度判断：

- 单选、多选、空答、异常答案、多选部分得分已有核心规则。
- 判断题没有独立产品化题型，需要明确用 A/B、对/错、T/F 还是 true/false，并补测试。
- 修改答案后重新判分从流程上可通过重新确认答案和批改实现，但 UI 体验还不够直接。

### 检查合格

批改前必须检查：

1. 是否有学生；
2. 是否有标准答案；
3. 是否有学生答案；
4. 题数是否一致；
5. 是否有异常选项；
6. 是否有空答；
7. 是否有重复学生；
8. 是否有缺失学生姓名；
9. 是否有分值异常；
10. 是否可以开始批改。

当前完成度判断：

- `legacy.objective_grader_legacy.build_validation_report` 和 `app.validators.has_blocking_errors` 已有异常检查基础。
- 重复题号、重复学生、空 submissions、异常答案等已有测试或流程覆盖。
- 面向老师的错误解释和“下一步该怎么修”还不够，需要产品化。

### 输出合格

必须输出：

1. 学生总分；
2. 每题得分；
3. 每题作答；
4. 错题号；
5. 每题正确率；
6. 班级平均分；
7. 最高分；
8. 最低分；
9. 错得最多的题；
10. 薄弱知识点；
11. Excel 成绩表；
12. 简洁班级报告。

当前完成度判断：

- summary、detail、item_analysis、knowledge_profile、student_wrong_list、teaching_plan、exam_report.xlsx 等已有。
- 输出内容较丰富，但“班级简报”作为第一层报告还需要强化，把老师最关心的结论集中展示。
- 学生个人报告已有 `student_report.csv` 基础，但还不够像“个人情况报告”。

### 使用体验合格

必须做到：

1. 按钮名字简单；
2. 步骤清楚；
3. 报错能看懂；
4. 不懂电脑的人也能照着做；
5. 会电脑的人能看到详细数据；
6. 不强迫老师理解技术术语；
7. 不让老师重复输入；
8. 不轻易丢失数据。

当前完成度判断：

- 当前已有网页向导雏形，这是很好的基础。
- 前端 `web/static/app.js` 集中承担状态、API、渲染、摄像头、表格编辑、批改流程，后续继续加产品化体验会加重维护风险。
- 数据落地到 `data/`，不完全依赖浏览器内存，这是优点；但草稿保存、误关闭恢复、编辑后重新批改体验仍需强化。

## 4. 当前功能完成度审计

### 已经够用或接近够用的功能

- 本地网页端已经存在，老师不必只用命令行。
- 班级导入和班级索引已经存在。
- 考试创建流程已有网页向导。
- 标准答案导入来源较多，包括表格、文本、Word/PDF、图片占位入口。
- 学生作答表格导入已经存在。
- 批改前预览和数据检查已经存在。
- 自动判分核心已经存在。
- 多选部分得分规则已经实现。
- 班级统计、题目分析、知识点分析、讲评计划、Excel 报告已经有雏形。
- 测试入口 `python run_tests.py` 已覆盖一批核心规则和文件流程。

### 还不够老师真实使用的功能

- 学生名单手动新增、修改、删除不够产品化。
- 标准答案手动录入虽然有草稿表格，但对不懂电脑老师仍可能偏复杂。
- 学生答案手动录入或粘贴录入能力不够清晰。
- 判断题作为独立题型的展示、录入、判分和测试不够明确。
- 批改前异常检查需要从“技术校验结果”变成“老师修复清单”。
- 班级简报还需要更聚焦，先给结论。
- 学生个人情况需要更像老师能直接看懂的个人报告。
- Excel 导出需要按老师实际使用拆成清晰 Sheet。
- 数据草稿和历史恢复机制需要增强，避免老师误操作后重来。

### 对不懂电脑老师偏复杂的步骤

- 标准答案来源方式较多，容易让老师犹豫，不如默认提供“手动输入 / Excel 导入”两个主入口，其余放更多方式。
- 学生作答导入需要老师理解表头格式，如 `Q1`、`Q2`，需要更明显模板和示例。
- 检查数据阶段如果只显示错误列表，老师可能不知道怎么修。
- 报告中心如果先展示多个报告文件链接，老师可能不知道先看哪个。

### 对会电脑老师分析不够细的地方

- 每题选项分布已有数据，但需要更清晰地展示和导出。
- 学生个人薄弱点需要更明确排名、错题、知识点、建议动作。
- 知识点统计需要区分班级层面和学生层面。
- Excel 需要稳定 Sheet 和字段说明。

## 5. 批改主流程设计

推荐最简单主流程：

选择班级 -> 新建考试 -> 录入答案 -> 导入学生作答 -> 检查数据 -> 开始批改 -> 查看报告 -> 导出成绩。

### 选择班级

页面文案：“这次批改哪个班？”

动作：

- 选择已有班级；
- 新建班级；
- 导入学生名单；
- 查看学生名单是否正确。

### 新建考试

页面文案：“这次考试叫什么？”

字段：

- 考试名称；
- 考试日期；
- 科目；
- 总分，可选；
- 备注，可选。

### 录入答案

页面文案：“把正确答案放进来。”

默认方式：

- 手动输入答案；
- 从 Excel 导入。

后续方式：

- 从题库导入试卷，暂缓；
- 图片识别答案，暂缓。

### 导入学生作答

页面文案：“把学生作答放进来。”

方式：

- Excel/CSV 导入；
- 粘贴表格；
- 手动录入，后续补齐；
- 拍照识别，暂缓。

### 检查数据

页面文案：“先检查有没有问题。”

显示：

- 学生人数；
- 题目数量；
- 标准答案是否完整；
- 学生答案是否完整；
- 异常选项；
- 空答；
- 未匹配学生；
- 是否可以开始批改。

### 开始批改

主按钮：“开始批改”。

批改过程不允许 AI/OCR 介入判分，只调用规则判分。

### 查看报告

默认打开“班级简报”。

二级入口：

- 查看详细分析；
- 查看学生情况；
- 导出成绩表。

### 导出成绩

主按钮：“导出成绩表”。

详细入口：

- 导出完整分析 Excel；
- 导出班级简报；
- 导出学生个人情况。

## 7. 报告体系设计

### 班级简报

目标用户：不懂电脑或只想快速看结果的老师。

内容：

1. 本次考试人数；
2. 平均分；
3. 最高分；
4. 最低分；
5. 错得最多的 5 道题；
6. 最薄弱的 3 个知识点；
7. 需要重点关注的学生；
8. 建议优先讲评的题。

设计原则：

- 一个屏幕先看懂；
- 少表格，多结论；
- 每条结论都能点进详细数据；
- 默认报告就是它。

### 详细分析

目标用户：会电脑、愿意深入看的老师。

内容：

1. 每题正确率；
2. 每题得分率；
3. 每题选项分布；
4. 每个学生错题；
5. 每个学生薄弱知识点；
6. 知识点得分率；
7. 分数段分布；
8. Excel 明细导出。

设计原则：

- 表格字段稳定；
- 支持按题目、学生、知识点查看；
- 支持导出；
- 不影响班级简报的简洁性。

### 学生个人情况

内容：

1. 学生总分；
2. 班级位置或排名；
3. 错题列表；
4. 空答和异常答案；
5. 薄弱知识点；
6. 与班级平均相比；
7. 后续练习建议。

当前系统有学生报告 CSV 基础，但需要产品化成老师能直接阅读的“学生个人情况”。

### Excel 导出

Excel 是老师真实工作中的关键出口。导出不应只是机器数据，而应有清晰 Sheet、清晰列名和可直接使用的汇总。

## 8. 异常检查设计

批改前检查应分为“必须修复”和“建议关注”。

### 必须修复

- 没有学生名单；
- 没有标准答案；
- 没有学生作答；
- 标准答案题号重复；
- 学生编号重复且无法区分；
- 学生作答题号与标准答案严重不匹配；
- 标准答案缺少关键题答案；
- 分值为空或非法。

老师看到的提示示例：

- “第 3 题没有标准答案，请补充后再批改。”
- “学生名单里有重复学号 230301，请先修改。”
- “学生答案文件里没有找到 Q1、Q2 这类题目列。”

### 建议关注

- 某学生有空答；
- 某题空答人数较多；
- 某学生答案含异常选项；
- 作答文件中出现标准答案没有的题号；
- 学生姓名缺失但学号存在；
- 班级名单中有学生没有作答。

老师看到的提示示例：

- “张三第 5 题为空答，系统会按 0 分处理。”
- “李四第 2 题填写了 E，但本题只允许 A-D，请检查。”
- “有 3 名学生在名单中，但没有提交答案。”

### 检查页设计

检查页应显示：

- 是否可以开始批改；
- 必须修复项数量；
- 建议关注项数量；
- 学生人数；
- 题目数量；
- 可批改题数量；
- 主要修复按钮或指引。

## 10. 当前暂缓功能

以下功能当前全部暂缓：

1. 题库接入；
2. `paper-package.json`；
3. `assessment-result.json`；
4. 题库回流；
5. OCR；
6. AI；
7. 自动组卷；
8. AI 生成练习；
9. 复杂图表大屏；
10. 多系统耦合。

暂缓原因：

- 当前批改主链仍需产品化；
- 老师使用体验还需简化；
- 判分核心仍在 legacy；
- 前端单文件职责偏多；
- 报告需要先分层；
- 过早接 OCR/AI 会增加数据污染风险；
- 过早接题库会把不稳定内部模型暴露成跨系统协议。

## 11. 工程结构风险

### legacy

`legacy/objective_grader_legacy.py` 仍承担模型、判分、统计、报告、HTML、Excel、校验等大量职责。继续把核心功能加进去会让系统越来越难维护。近期应避免新增 legacy 核心逻辑。

### app/core.py

`app/core.py` 当前主要 re-export legacy 中的 `AnswerKey`、`QuestionSpec`、`StudentResult`、`score_answer`、`grade_all` 等。它还不是真正独立的判分核心。

### app/workflow.py

`app/workflow.py` 是当前批改流程编排中心，负责读取、判分、校验、生成报告、归档等。它对产品闭环很关键，但职责也偏重。后续应逐步拆出 use case、report、exporter。

### web/static/app.js

`web/static/app.js` 同时承担全局状态、API 调用、DOM 渲染、页面导航、摄像头、文件上传、表格编辑、批改流程控制、报告展示。继续扩展 UI 会让风险增加。后续应拆成 state、api、wizard、tables、reports、camera 等模块。

### 报告生成

报告生成散布在 legacy 和 `app/workflow.py` 中，HTML、CSV、Excel 混在流程里。后续产品化报告时，应先定义报告数据模型，再由不同 exporter 输出 HTML/Excel/CSV。

### Excel 导出

当前已有 Excel 输出能力，但需要稳定面向老师的 Sheet 结构。Excel 导出不应直接依赖零散中间 CSV 文件名，而应依赖稳定的报告数据对象。

### 测试覆盖

当前 `python run_tests.py` 覆盖 13 个测试，已有基础。但产品化前还需要补：

- 判断题；
- 多选 3 个正确答案的部分得分；
- 修改答案后重新判分；
- 异常答案提示；
- Excel Sheet 字段；
- 班级简报数据；
- 学生个人情况数据。

## 12. 下一阶段开发路线

### Stage 1：判分核心稳定

目标：确保单选、多选、判断题、空答、异常答案规则可靠。

范围：

- 明确判分规则；
- 补测试；
- 不改变现有行为；
- 为迁出 legacy 做准备。

验收标准：

- 单选正确/错误测试通过；
- 多选全对、漏选、含错选、空答测试通过；
- 判断题规则明确并有测试；
- 异常答案返回明确状态；
- `python run_tests.py` 通过。

### Stage 2：批改主流程产品化

目标：让老师按步骤完成一次批改。

范围：

- 选择班级；
- 新建考试；
- 录入答案；
- 导入学生作答；
- 检查数据；
- 开始批改；
- 查看报告；
- 导出成绩。

验收标准：

- 新老师能跟着向导完成；
- 每一步有下一步提示；
- 不出现技术术语；
- 不需要重复输入同一信息。

### Stage 3：异常检查完善

目标：批改前发现数据问题。

范围：

- 必须修复项；
- 建议关注项；
- 老师可读错误；
- 修复指引。

验收标准：

- 缺标准答案、缺学生作答、题号不一致、异常选项、重复学生均能提示；
- 错误提示能说明怎么修；
- 阻塞项不能直接批改，除非老师明确试算。

### Stage 4：报告体系完善

目标：班级简报 + 详细分析 + 学生个人情况。

范围：

- 班级简报；
- 详细分析；
- 学生个人情况；
- 讲评建议。

验收标准：

- 班级简报一屏可读；
- 详细分析包含每题、每学生、每知识点；
- 学生个人情况能说明错题和薄弱点；
- 讲评建议能回答先讲什么。

### Stage 5：Excel 导出完善

目标：导出老师真正能用的成绩表和分析表。

范围：

- 学生成绩总表；
- 每题统计；
- 学生错题；
- 知识点统计；
- 原始作答。

验收标准：

- Sheet 名称清楚；
- 字段稳定；
- 老师打开即可使用；
- 原始作答可追溯；
- 导出测试通过。

### Stage 6：UI 简化美化

目标：减少术语，强化大按钮和向导。

范围：

- 首页；
- 新建批改向导；
- 检查页；
- 报告入口；
- 导出入口。

验收标准：

- 普通老师主流程不出现专业术语；
- 主按钮清楚；
- 页面不拥挤；
- 复杂功能收进详细分析或更多设置。

### Stage 7：再考虑题库接入

目标：在批改软件合格后，再做数据接入。

范围：

- 题库试卷导入；
- 题库回流导出；
- 稳定 questionId。

验收标准：

- 批改主链已稳定；
- 判分和报告模型已稳定；
- 题库接入不影响普通批改流程；
- 不引入 OCR/AI 依赖。

## 13. 最近一步建议

根据当前代码真实情况，下一步建议执行：

Stage 1：判分核心稳定 + 测试补齐。

理由：

1. 当前判分规则已经可用，但核心仍在 legacy。
2. 单选、多选、空答、异常答案已有基础测试。
3. 判断题还没有作为独立题型明确。
4. `run_tests.py` 对多选 3 个正确答案的比例得分覆盖不足。
5. 在批改软件产品化前，必须先确认“分数算得对”。

Stage 1 不应马上大迁移。建议先做最小工作：

- 补判分规则说明文档或测试用例；
- 补判断题测试；
- 补多选 3 正确答案测试；
- 补空答和异常答案更完整测试；
- 保持现有行为不变；
- 测试稳定后，再考虑把判分核心从 legacy 迁出。

如果 Stage 1 完成且测试稳定，再进入 Stage 2：批改主流程产品化。
